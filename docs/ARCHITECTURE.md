## Overview

`benisty.sh` is a personal blog/portfolio site built around a fully **static, serverless, AWS-backed** architecture. It uses [Hugo](https://gohugo.io/) (extended) as the static site generator, the third-party [FixIt](https://github.com/hugo-fixit/FixIt) theme as a base UI framework, and a stack of **AWS S3 + CloudFront + CloudFlare + GitHub Actions** for hosting, distribution, security and deployment.

The repository is essentially three things glued together:

1. A Hugo project (content + custom layout overrides + SCSS overrides on top of the FixIt theme).
2. A small piece of AWS infrastructure described in Terraform (`cloudfront.tf`).
3. A GitHub Actions deployment pipeline (`.github/workflows/push_to_s3.yaml`) that ships every push to `main` to production.

The site itself documents this exact architecture in the article `content/posts/this-site-architecture.md`.

## 1. Static Site Generator: Hugo + FixIt

### Why Hugo

Hugo is a Go-based static site generator. The site picks it because:

- It produces a folder of plain HTML/CSS/JS — a perfect fit for S3 + CloudFront.
- It has a strong theme ecosystem (FixIt is used here) so the front-end framework is essentially "free".
- Its build is one command (`hugo --minify`) with no Node toolchain required at runtime.

### Theme: FixIt (as a Git submodule)

`.gitmodules`:

```1:3:.gitmodules
[submodule "themes/FixIt"]
	path = themes/FixIt
	url = https://github.com/hugo-fixit/FixIt.git
```

The theme is referenced from `hugo.toml` via `theme = 'FixIt'`. By keeping FixIt as a submodule:

- The repo stays small and easy to update (`git submodule update --remote`).
- The CI runner re-clones the theme on every build, ensuring reproducible deploys.

### Site Configuration — `hugo.toml`

The configuration file wires together theme behavior, parameters and the navigation menu. Highlights:

```1:14:hugo.toml
baseURL = 'https://benisty.sh/'
languageCode = 'en-us'
title = 'benisty.sh'
theme = 'FixIt'

[outputs]
  home = ['html', 'rss', 'search']

[markup]
  _merge = 'shallow'
  [markup.goldmark]
    [markup.goldmark.renderer]
      unsafe = true
```

Key technical decisions visible here:

- **`outputs.home`** generates not just `index.html`, but also an RSS feed and a `search` JSON index used by the FixIt search dropdown.
- **`markup.goldmark.renderer.unsafe = true`** allows raw HTML inside markdown — this is what makes the inline `<img>` tags with custom `style` attributes in posts (see `this-site-architecture.md`) actually render.
- **`[markup] _merge = 'shallow'`** and **`[taxonomies] _merge = 'shallow'`** instruct Hugo to merge these tables shallowly with the theme's defaults instead of replacing them, so taxonomy/markup defaults from FixIt are preserved.

The `[params]` block configures FixIt features without touching the theme code:

- **`defaultTheme = 'dark'`** — forces the site into dark mode by default.
- **`[params.app]`** — favicon paths, PWA tile/icon colors (`iconColor = '#4ade80'`, a green accent that matches the dark UI).
- **`[params.search]`** — enables client-side search using **Fuse.js**:

  ```31:38:hugo.toml
    [params.search]
      enable = true
      type = 'fuse'
      contentLength = 4000
      placeholder = 'Search...'
      maxResultLength = 10
      snippetLength = 30
      highlightTag = 'em'
  ```

  Hugo emits a JSON index at build time (because of `home = [..., 'search']`); FixIt loads it client-side and uses Fuse to do fuzzy searching, with no backend required. `contentLength` / `snippetLength` cap how much body text is indexed and shown per hit.

- **`[params.home.profile]`** — the homepage hero/profile card content: avatar URL, title, terminal-styled subtitle (`<span style="color:#555">$</span> <span style="color:#aaa">whoami</span> ... Linux Enthusiast`), `typeit = true` (typewriter animation via the FixIt-bundled TypeIt plugin), and the `techStack` list rendered as a scrolling marquee.
- **`[params.home.posts]`** — homepage post listing: `paginate = 6`, plus a `featuredPosts` array that pins `this-site-architecture.md` at the top of the homepage with a special "Featured" card.
- **`[params.codeblock]`** — `mode = "simple"` switches FixIt's code blocks to its lightweight renderer (no copy buttons / line-number gutter scaffolding), and `maxShownLines = 0` disables the "show more" collapse so code blocks render in full.
- **`[params.page.share]`** — enables Twitter / Facebook / LinkedIn share buttons on individual posts.

The `[menus]` block defines the four header items (`Posts`, `GitHub`, `LinkedIn`, `Email`) with FontAwesome icons supplied via `[menus.main.params].icon`.

### SCSS Overrides — Hugo Pipes integration

`assets/css/` is processed by Hugo's asset pipeline. FixIt's own SCSS imports these files by convention:

- **`_override.scss`** — Sass variable overrides loaded **before** FixIt's own variable definitions, so they win. It defines a clean dark palette (`$global-background-color: #111111`, `$single-link-color: #6ab0f3`, `$code-background-color: #1a1a1a`, `$scrollbar-color: #333333`, etc.). This is the proper, theme-aware way to recolor FixIt — no `!important` needed because variables propagate through every component's stylesheet.

- **`_custom.scss`** — appended at the end of FixIt's pipeline, used for arbitrary site-specific CSS rules (and where `!important` is liberally used to defeat the theme's own selectors). It contains:

  - **Hidden elements**: removes things FixIt enables by default but the site doesn't want — mobile header, theme switcher, breadcrumbs, the "delimiter" menu separator.
  - **Header**: makes `#header-desktop` `position: absolute; background: transparent` **only on the homepage**, and falls back to a relative, opaque `#111` header on every other page using a CSS `:has()` selector:

    ```73:76:assets/css/_custom.scss
    body:not(:has(.page.home)) #header-desktop {
      position: relative !important;
      background-color: #111 !important;
    }
    ```

    This is a modern CSS trick that lets the entire header switch behavior based on what's inside `<body>` without any JS.

  - **Homepage hero**: `.home-profile { height: 100vh; }` makes the hero exactly one screen tall, positioning the scroll indicator and post list as a "second fold".
  - **Profile card, avatar, subtitle (JetBrains Mono terminal look), disclaimer, resume button, social icons, summary cards, featured-post cards, tech strip, scroll indicator** — all hand-styled.
  - **Search dropdown**: hidden when empty using another `:has()` selector so the empty floating dropdown doesn't appear on focus.
  - **Responsive**: a single mobile breakpoint at `680px` shrinks the avatar / title / subtitle.

- **`assets/jsconfig.json`** is purely an editor convenience — it tells JS-aware editors to resolve bare imports against `themes/FixIt/assets/*` so IntelliSense works when reading the theme's JS sources from project-level overrides.

## 2. Hosting & CDN: AWS S3 + CloudFront

### S3 (Origin)

The S3 bucket is the source of truth for the built site. It is configured as **private** (block public access on) — the bucket is **not** exposed via S3 Website Hosting endpoints. Instead, CloudFront accesses it via an **Origin Access Identity (OAI)** so only signed CloudFront requests can read objects.

The bucket policy (from the `this-site-architecture.md` article) has two statements:

1. `AllowCloudFrontServicePrincipal` — `Allow s3:GetObject` to the CloudFront service principal **conditioned** on `AWS:SourceArn` matching this specific distribution. This prevents other people's CloudFront distributions from reading the bucket.
2. `DenyInsecurePolicy` — explicit `Deny` on `s3:*` for any request where `aws:SecureTransport == false`, enforcing HTTPS-only access.

### CloudFront (Edge / CDN) — defined in `cloudfront.tf`

The Terraform file declares three resources:

1. `aws_cloudfront_origin_access_identity.cloudfront` — the OAI used by the bucket policy.

2. `aws_cloudfront_distribution.website` — the main distribution. Notable settings:

   ```16:40:cloudfront.tf
     default_cache_behavior {
       allowed_methods  = ["GET", "HEAD", "OPTIONS"]
       cached_methods   = ["GET", "HEAD", "OPTIONS"]
       target_origin_id = "s3-website"

       forwarded_values {
         query_string = false
         headers      = []

         cookies {
           forward = "none"
         }
       }

       viewer_protocol_policy = "redirect-to-https"
       min_ttl                = 0
       default_ttl            = 86400
       max_ttl                = 31536000
       compress               = true

       function_association {
         function_arn = aws_cloudfront_function.cache_key_function.arn
         event_type   = "viewer-request"
       }
     }
   ```

   - **`viewer_protocol_policy = "redirect-to-https"`** — HTTP requests are 301-redirected to HTTPS at the edge.
   - **`compress = true`** — CloudFront serves gzip/brotli automatically.
   - **`default_ttl = 86400`, `max_ttl = 31536000`** — default of 1 day, max of 1 year, so static assets are aggressively cached. Cache invalidation is handled in CI (see below).
   - **No query strings, no headers, no cookies forwarded** — every request collapses to one cache key per URI, maximizing cache hit rate (a static site doesn't vary by any of those).
   - **Viewer-request function**: a CloudFront Function (cheaper and faster than Lambda@Edge) is associated to rewrite incoming URIs.
   - `default_root_object = "index.html"` — bare `/` requests are served the home page automatically.
   - `price_class = "PriceClass_100"` — only deploys to NA + EU edges, the cheapest tier.
   - `geo_restriction.restriction_type = "none"` — no built-in CloudFront geo blocking (geo blocking is delegated to CloudFlare WAF).
   - `viewer_certificate.cloudfront_default_certificate = true` — uses the default `*.cloudfront.net` certificate. (The article notes that ACM is used for HTTPS on the custom domain; that part of the certificate config isn't versioned here, so the Terraform you see represents the simplest working baseline.)

   There is also an **`ordered_cache_behavior` for `path_pattern = "/*"`** with shorter TTLs (`default_ttl = 3600`, `max_ttl = 86400`). Because `/*` matches everything, this behavior in practice covers all paths and effectively becomes the operative caching policy for the site, providing a more aggressive 1-hour default freshness window for HTML pages while the `default_cache_behavior` catches anything that somehow doesn't match.

3. `aws_cloudfront_function.cache_key_function` — a JavaScript function (CloudFront Functions runtime, not Lambda) that runs at the **viewer-request** stage on every request:

   ```86:105:cloudfront.tf
   resource "aws_cloudfront_function" "cache_key_function" {
     name    = "benisty.sh-cache-key"
     runtime = "cloudfront-js-1.0"
     type    = "VIEWER_REQUEST"

     code = <<EOF
   function handler(event) {
     var request = event.request;
     var uri = request.uri;

     if (uri.endsWith('/')) {
       request.uri = uri + 'index.html';
     } else if (!uri.includes('.')) {
       request.uri = uri + '/index.html';
     }

     return request;
   }
   EOF
   }
   ```

   This is the critical glue between Hugo's "pretty URLs" and S3's literal-key behavior:

   - Hugo writes pages to `posts/this-site-architecture/index.html`.
   - A user requests `/posts/this-site-architecture` (no trailing slash, no extension). S3 only knows about the literal key — it would 404.
   - The viewer-request function rewrites `/posts/this-site-architecture` → `/posts/this-site-architecture/index.html`, and `/posts/this-site-architecture/` → `/posts/this-site-architecture/index.html`.
   - Anything containing a `.` (e.g. `/images/avatar.jpg`) is treated as a real file and passed through untouched.

   CloudFront Functions are sub-millisecond and cost ~$0.10 per million invocations, so this is essentially free.

### CloudFlare (DNS + WAF)

Although there is no committed CloudFlare configuration in the repo, the article `this-site-architecture.md` documents the actual setup:

- The domain's authoritative nameservers point to CloudFlare.
- A DNS record (proxied — orange cloud) points `benisty.sh` → CloudFront distribution.
- TLS mode is set to **Full** so CloudFlare always speaks HTTPS to CloudFront.
- An "always use HTTPS" rule redirects all HTTP at the edge.
- A page rule **disables CloudFlare's caching** for the entire domain — caching is handled by CloudFront, and double-caching causes invalidation pain.
- WAF rules block traffic from a list of high-bot-traffic countries.

The CI pipeline still issues a CloudFlare cache purge on every deploy in case CloudFlare caching is ever re-enabled.

## 3. CI/CD: GitHub Actions — `.github/workflows/push_to_s3.yaml`

A single workflow handles the entire deploy on every push to `main`:

```1:46:.github/workflows/push_to_s3.yaml
name: Deploy Hugo Site to S3

on:
  push:
    branches:
      - main 

jobs:
  build-and-deploy:
    runs-on: ubuntu-latest
    
    steps:
      - name: Checkout Code
        uses: actions/checkout@v4
        with:
          submodules: true

      - name: Setup Hugo
        uses: peaceiris/actions-hugo@v3
        with:
          hugo-version: 'latest'
          extended: true 

      - name: Build
        run: hugo --minify

      - name: Configure AWS Credentials
        uses: aws-actions/configure-aws-credentials@v4
        with:
          aws-access-key-id: ${{ secrets.AWS_ACCESS_KEY_ID }}
          aws-secret-access-key: ${{ secrets.AWS_SECRET_ACCESS_KEY }}
          aws-region: il-central-1 

      - name: Deploy to S3
        run: aws s3 sync ./public s3://${{ secrets.S3_BUCKET_NAME }} --delete

      - name: Invalidate CloudFront Cache
        run: aws cloudfront create-invalidation --distribution-id ${{ secrets.AWS_CLOUDFRONT_DISTRIBUTION }} --paths "/*"

      - name: Purge Cloudflare Cache
        run: |
          curl -s -X POST "${{ secrets.CLOUDFLARE_CACHE_ENDPOINT }}" \
            -H "Authorization: Bearer ${{ secrets.CLOUDFLARE_API_TOKEN }}" \
            -H "Content-Type: application/json" \
            --data '{"purge_everything":true}'
```

Walking through it stage by stage:

1. **Trigger**: push to `main`. There's no PR-preview / staging environment — small personal site, edits go straight from `main` to production.

2. **Checkout with submodules**: `submodules: true` is essential because the FixIt theme lives in a submodule. Without it, `themes/FixIt` would be empty and Hugo would fail.

3. **Setup Hugo** via `peaceiris/actions-hugo@v3` with `extended: true`. The **extended** edition is required because `_override.scss` and `_custom.scss` use **SCSS** (Sass), and only `hugo extended` ships with the `libsass`/Dart-Sass transpiler.

4. **Build**: `hugo --minify`. Hugo writes the entire site to `./public/` and minifies HTML / CSS / JS / JSON / SVG output in place.

5. **AWS credentials**: configured via repository secrets, scoped to the `il-central-1` (Tel Aviv) AWS region.

6. **S3 deploy**: `aws s3 sync ./public s3://$BUCKET --delete`. The `--delete` flag is what makes this an **atomic-ish, idempotent deploy**: any object in the bucket that's no longer in the build output is removed, so renamed/deleted posts disappear from the site. Sync compares sizes and timestamps so unchanged objects are skipped — only the diff is uploaded.

7. **CloudFront invalidation**: `aws cloudfront create-invalidation ... --paths "/*"`. Because the `default_ttl` is 1 day / 1 hour, freshly deployed pages would otherwise be served stale from edge caches. A blanket `/*` invalidation forces every edge to re-fetch on the next request. AWS gives 1,000 invalidation paths/month free (and `/*` counts as a single path), so this is also free in practice.

8. **CloudFlare cache purge**: `curl -X POST` to the CloudFlare cache-purge endpoint with `{"purge_everything": true}`. This is a defensive step (caching is currently disabled at CloudFlare) so future re-enables don't cause stale content.

There is no test/lint stage — for a static blog with no app code, the build itself is the validation: if Hugo can compile the site, it deploys.

## 4. End-to-End Request Lifecycle

### Content publishing

```
Author edits content/posts/foo.md
        ↓
git push origin main
        ↓
GitHub Actions: checkout (with FixIt submodule)
        ↓
Hugo extended builds → public/  (HTML/CSS/JS minified)
        ↓
aws s3 sync ./public s3://<bucket> --delete   (idempotent diff upload)
        ↓
aws cloudfront create-invalidation /*         (drop edge caches)
        ↓
curl POST CloudFlare purge_everything         (defensive)
```

The author's job ends at `git push`. Time from push to live edge is ~30–60s.

### User request

```
Browser → benisty.sh
        → CloudFlare DNS resolves and proxies (WAF + geo block + HTTPS upgrade)
        → CloudFront edge
              → CloudFront Function (viewer-request) rewrites URI to .../index.html
              → Cache lookup
                    HIT  → return cached body (gzip/brotli, HTTPS)
                    MISS → fetch from S3 origin via OAI (S3 enforces CF source ARN)
                         → cache and return
        ← user gets HTML, then static asset requests follow the same path
```

Typical request latency is dominated by the CloudFlare → CloudFront hop and the TLS handshake; once warm, edge cache hits are sub-50ms.

## 5. Cost & Operational Characteristics

The architecture is deliberately **free-tier-friendly**:

| Component        | Cost                                                                |
| ---------------- | ------------------------------------------------------------------- |
| S3               | 5 GB / 20K GET on free tier — site is < 100 MB                      |
| CloudFront       | 1 TB transfer + 10M requests/mo on free tier                        |
| CloudFront Funcs | $0.10 per million invocations                                       |
| CloudFlare       | DNS + WAF free                                                      |
| GitHub Actions   | Free for public repos / 2000 min/mo for private                     |
| Domain           | ~$3/year                                                            |

Effective recurring cost: roughly **$1/month** amortized.

Operationally:

- **No servers to patch** — pure managed services + static files.
- **Atomic-ish deploys** via `s3 sync --delete` — the only window where a user might see a half-deployed state is between the first uploaded file and the cache invalidation, and even then they're served the cached previous version.
- **Strong cache layering**: CloudFront is the only caching layer (CloudFlare cache disabled), invalidated on each deploy, with the viewer-request function transparently fixing pretty URLs.
- **Security model**: bucket fully private; only the specific CloudFront distribution can read; HTTPS enforced both at the bucket policy and at CloudFront; WAF + geo blocks at CloudFlare; no secrets in the repo (all via GitHub Actions secrets).

## 6. Summary of Key Technical Decisions

- **Hugo (extended) + FixIt theme as submodule** — minimal, reproducible, no Node toolchain at build time.
- **Theme customization without forking** — `assets/css/_override.scss` for variable-level theming, `assets/css/_custom.scss` for ad-hoc rules, and `layouts/` for template overrides (`home.html`, `_partials/home/profile.html`).
- **Custom homepage** with full-viewport profile hero, animated tech-stack marquee, scroll-down affordance, and pinned featured posts driven by `params.home.posts.featuredPosts` in `hugo.toml`.
- **S3 private origin + CloudFront OAI** instead of S3 website hosting — keeps the bucket private and forces all reads through CloudFront where TTLs, compression and edge logic apply.
- **CloudFront Functions (viewer-request)** to map Hugo's pretty URLs to S3 keys — sub-ms, near-free, no Lambda cold-start.
- **CloudFlare for DNS + WAF only**, CloudFront for caching — no double-cache, clear separation of concerns.
- **GitHub Actions pipeline**: checkout (with submodules) → Hugo build → `s3 sync --delete` → CloudFront invalidation → CloudFlare purge. Simple, fast, idempotent.
- **Infrastructure as code** for the CDN layer (`cloudfront.tf`) so the distribution + OAI + viewer-request function are reproducible.

The whole project is a textbook example of "do as much as possible at the edge, store as little as possible at the origin, and let CI do the rest" for a static personal site.
